## learn-claude-code

### 1. 为什么要学这个教程？（打破幻觉）

市面上大多数关于 Agent 的讨论都停留在“提示词工程（Prompt Engineering）”或“低代码工作流”上。这个教程的核心观点是：**你可能一直在错误的地方努力。**

*   **纠正错误认知：** 很多人认为 Agent 的智能来自于复杂的 `if-else` 逻辑或流程图。该教程明确指出：**Agency（感知、推理、行动的能力）来自模型训练，而不是代码编排。**
*   **拒绝“水管工”式开发：** 教程抨击了那种把 LLM 当作补全节点、用硬编码路由连接的“鲁布·戈德堡机械”。这种系统脆弱且不可扩展。
    *   一个过度工程化的、脆弱的过程式规则流水线，LLM 被楔在里面当一个美化了的文本补全节点。那不是 Agent。那是一个有着宏大妄想的 shell 脚本。

*   **回归本质：** 学习如何像 Claude、GPT、Gemini  那样思考 Agent，即：**模型（驾驶者）+ Harness（载具）**。

### 2. 你能收获什么？（从“编排者”转变为“Harness 工程师”）

通过这个教程，你将学会构建一个专业的 **Harness（载具/基础设施）**，这是让模型智能得以释放的“物理世界”。

*   **12 个递进的工程模式：** 从最简单的循环（s01）到复杂的隔离执行环境（s12），你将掌握构建生产级 Agent 的全套架构。
*   **核心工程能力：**
    *   **工具实现：** 如何给 Agent 一双手（文件、Shell、浏览器）。
    *   **上下文管理：** 学习三层压缩策略，解决 Token 溢出问题。
    *   **任务拆解与持久化：** 学习如何让 Agent 拥有长期记忆和复杂的任务依赖图。
    *   **团队协作：** 掌握多 Agent 之间的异步通信与自组织认领机制。
    *   **环境隔离：** 学习使用 `worktree` 等技术确保 Agent 在操作时的安全性与独立性。

### 3. 教程的内部思考逻辑是什么？（核心心智模型）

该教程的逻辑可以拆解为以下三个层面：

#### A. 公式化定义
$$Harness = Tools + Knowledge + Observation + Action \ Interfaces + Permissions$$​
教程认为，工程师的工作不是去“写”智能，而是去“造”一个环境。环境越清晰、工具越原子化、权限越明确，模型的 Agency 表达就越有效。

```
		Tools:          文件读写、Shell、网络、数据库、浏览器
    Knowledge:      产品文档、领域资料、API 规范、风格指南
    Observation:    git diff、错误日志、浏览器状态、传感器数据
    Action:         CLI 命令、API 调用、UI 交互
    Permissions:    沙箱隔离、审批流程、信任边界
```

#### B. “信任模型”的哲学
教程以 **Claude Code** 为标本，其核心逻辑是：**不要试图替模型做决定。**
*   不要用僵化的工作流限制它。
*   给它工具，给它知识，给它边界，然后**让开路**，让模型自己去推理和执行。

#### C. 递进式的演进路径
教程的结构反映了 Agent 从“玩具”到“工具”再到“员工”的演进过程：
1.  **第一阶段（循环）：** 建立最基本的“感知-行动”闭环（s01-s02）。
2.  **第二阶段（规划）：** 引入 Todo 机制和子 Agent，解决“走哪算哪”的问题（s03-s06）。
3.  **第三阶段（持久化）：** 让 Agent 能够处理跨越会话的长短期任务（s07-s08）。
4.  **第四阶段（团队）：** 模拟人类组织架构，实现多 Agent 协作与资源隔离（s09-s12）。

## 原生 Agency 派vs图编排工作流派

### 1. 核心矛盾：谁拥有“控制权”？

*   **LangGraph（工作流派）：** 核心逻辑是 **“受控的自由”**。开发者通过定义状态机（Nodes 和 Edges）来规定 Agent 的思考路径。它解决的是“可靠性”问题，适用于业务逻辑极其复杂、容错率低、需要严格遵守合规流程的场景（如：金融审批、法律合规检查）。
*   **本文观点（Harness 派）：** 核心逻辑是 **“完全的授权”**。开发者只提供工具箱和沙箱，把决策权彻底交给模型。它解决的是“通用性”和“创造力”问题，适用于解决开放式、非标准化的任务（如：软件工程、自主研究）。

### 2. 对比理解：载具 vs 轨道

我们可以用交通系统来做类比：

| 维度         | Harness 模式 (本文)                                 | LangGraph 模式 (工作流)               |
| :----------- | :-------------------------------------------------- | :------------------------------------ |
| **类比**     | **越野车 (Off-road Vehicle)**                       | **有轨电车 (Tram)**                   |
| **环境**     | 荒野、未知的代码库、复杂的 Bug                      | 铺设好的轨道、标准的业务流程          |
| **优势**     | 极强的泛化能力，能处理开发者没想到的情况            | 极高的确定性，每一步都在监控和预期内  |
| **劣势**     | 可能会“绕路”或陷入死循环（需要 s06/s07 的机制约束） | 遇到轨道之外的情况就会卡死或报错      |
| **智能来源** | 几乎全部依赖模型自身的推理能力                      | 依赖模型推理 + 开发者硬编码的逻辑拓扑 |

### 3. 如何兼容：分层架构的艺术

在实际的生产环境中，你不需要二选一。最强大的系统往往是 **“在 Harness 中运行工作流，或在工作流中调用 Harness”**。

#### A. 宏观工作流 + 微观自主 (The "Hybrid" Pattern)
你可以使用 LangGraph 来定义宏观的业务阶段，但在每个阶段内部，启动一个本文所说的 Agent Loop。
*   **例子：** 一个审计系统。LangGraph 负责控制“数据抓取 -> 初审 -> 复审 -> 生成报告”的严格顺序。但在“初审”这个节点内部，它并不是简单的 Prompt，而是一个拥有 `read_file` 和 `bash` 工具的自主 Agent，它在给定的沙箱（Harness）里自由探索。

#### B. 把“工作流”作为一种“工具 (Tool)”
在本文的 Harness 模式下，你可以给模型提供一个工具叫做 `execute_standard_process`。
*   **逻辑：** 当模型意识到当前任务是一个标准流程（如：入职流程）时，它主动调用这个工具。这个工具内部可以是由 LangGraph 驱动的确定性工作流。

#### C. 动态 Harness 约束
LangGraph 的“图”本质上是给模型加了一层 **“认知约束”**。
*   你可以把 LangGraph 看作是一个更高级、更复杂的 **Harness 机制**。在本文的 s03（Todo）和 s07（任务系统）中，任务是动态生成的；而 LangGraph 则是把一部分任务结构“预先固化”了。

### 4. 总结：如何选择？

*   **如果你的任务目标是明确的，但路径是未知的**（例如：修复这个 Bug）：使用本文推荐的 **Harness 模式**。相信模型的 Agency，给它好用的工具和干净的环境。
*   **如果你的任务路径是必须合规的，且步骤是标准化的**（例如：银行开户）：使用 **LangGraph 模式**。通过图结构来补偿模型可能产生的随机性。

**最终的进化方向：**
优秀的工程师会构建一个**“具备工作流意识的 Harness”**。即：平时让模型自由发挥（s01-s12 的机制），但在关键决策点（如：涉及金钱交易、删除生产数据库），通过 Harness 层的代码强制介入，弹出一个审批流或切换到确定的逻辑分支。

这正是文章中提到的 **Permissions（权限治理）** 与 **Action Interfaces（行动接口）** 的高级进阶形态。
